# OOPs

- **Abstraction**: Model only attributes and behaviors relevant to the specific context (e.g., an Airplane in a flight simulator needs flight physics; in a ticket booking app, it only needs seat maps)
- **Encapsulation**: Hide the internal state. Use private to hide fields and expose a public interface (methods) for interaction. Clients should not depend on internal implementation details
- **Inheritance**: A subclass acts as an "Is-A" relationship. It reuses code from the parent but must technically suffice as a substitute for the parent
- **Polymorphism**: The program should detect the real class of an object and call its implementation, even when the type is unknown in the current context
- **Relations Constraints**:
    - **Dependency:** Object A uses Object B (transient).
    - **Association:** Object A refers to Object B (permanent link).
        - **Composition:** Object A *owns* Object B (lifecycle bound—e.g., University destroys its Departments).
        - **Aggregation:** Object A *has* Object B (lifecycle independent—e.g., Department has Professors)
