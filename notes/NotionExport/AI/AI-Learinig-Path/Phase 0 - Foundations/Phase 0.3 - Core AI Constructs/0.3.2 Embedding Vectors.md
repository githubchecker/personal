## 🗺️ Stage 0 — Concept Map

Tokens (0.3.1) let a model *read* text. **Embeddings** let a model *compare meaning*. An embedding
turns a piece of text into a list of numbers positioned so that similar meanings sit close
together. This is the single idea that makes RAG (Phase 2) possible: find relevant text by
geometry. Builds on tensors (0.2.1) and the embeddings milestone (0.2).

## 🎈 Stage 1 — The Simple Idea (analogy: a map of meaning)

Imagine a giant map where every sentence is a pin. Sentences about dogs cluster in one region,
sentences about finance in another. An **embedding** is the *coordinates* of a sentence's pin —
except instead of 2 numbers (latitude, longitude) it uses hundreds (e.g. 384, 768, or 1536).

**The "Aha!":** meaning becomes **location**. "Close on the map" = "similar in meaning." Searching
for relevant text becomes "find the nearest pins."

## ⚙️ Stage 2 — How It Actually Works

- **embedding** — a fixed-length vector (list of floats) representing a piece of text's meaning.
- **dimension** — how many numbers in the vector (e.g. 1536-D). More dimensions can capture more
  nuance but cost more storage.
- **cosine similarity** — compares the *angle* between two vectors: `1.0` = same direction
  (similar), `~0` = unrelated, `-1` = opposite. The standard semantic-search metric.

```python
from sentence_transformers import SentenceTransformer, util

model = SentenceTransformer("BAAI/bge-small-en-v1.5")   # produces 384-D vectors

# Each text -> one 384-number vector.
q = model.encode("How do I reset my password?", convert_to_tensor=True)
docs = model.encode([
    "Steps to recover a forgotten account login",   # related
    "Our refund policy for annual plans",            # unrelated
], convert_to_tensor=True)

# Compare the query to each doc by angle.
scores = util.cos_sim(q, docs)[0]
print(f"password vs login-recovery: {scores[0]:.3f}")   # higher
print(f"password vs refund-policy : {scores[1]:.3f}")   # lower
```

Why **cosine** over straight-line (Euclidean) distance: cosine ignores vector *length* and
compares only *direction* (the topic), which is what we care about for meaning. When vectors are
**normalised** (scaled to length 1), cosine similarity equals a simple **dot product** — a fast
shortcut used everywhere.

## 🚀 Stage 3 — In Practice / Why It Matters

The entire RAG pattern is: embed your documents once → store the vectors → embed each user
question → return the closest document vectors → feed that text to the LLM. Embeddings are also how
"semantic caching" (Phase 4) spots that two differently-worded questions mean the same thing.

**Most common beginner mistake:** mixing embedding models between documents and queries (also
flagged in 0.2.2). Different models = different maps; coordinates from one don't align with the
other, so "nearest" becomes meaningless. The fix: one embedding model for both sides.

## 🛑 STOP — Self-Check

Two sentences get cosine similarity `0.91`; another pair gets `0.12`. What does each number tell
you, and why might `0.91` happen even if the two sentences share no identical words?

<details><summary>Answer</summary>

`0.91` = very **similar in meaning** (vectors point almost the same direction); `0.12` = largely
**unrelated**. High similarity without shared words happens because embeddings encode *meaning*,
not spelling — synonyms and paraphrases land near each other on the "map," so their directions
align.
</details>
