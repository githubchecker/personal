## 🗺️ Stage 0 — Concept Map

LLMs don't read words — they read **tokens**. Understanding tokens explains three things you'll
meet constantly: **cost** (you pay per token), **context limits** (models hold a maximum number of
tokens), and **chunking** (Phase 2 splits documents by token count). This is the first Core AI
Construct and feeds directly into embeddings (0.3.2) and LLM APIs (0.3.3).

## 🎈 Stage 1 — The Simple Idea (analogy: LEGO bricks of text)

A model breaks text into **tokens** — small chunks that are often whole words but sometimes word
*pieces*. Think of text as built from LEGO bricks: common words are a single brick ("cat"), while
rare or long words snap together from several ("tokenization" → "token" + "ization").

**The "Aha!":** the model's whole world is a stream of these bricks. Everything — cost, memory,
limits — is counted in **tokens, not words or characters**.

Rough rule of thumb: **1 token ≈ 0.75 words**, so **1,000 tokens ≈ 750 words ≈ ~1.5 pages**.

## ⚙️ Stage 2 — How It Actually Works

The component that splits text is a **tokenizer**. OpenAI's is exposed via the `tiktoken` library.

Install: `pip install tiktoken`.

```python
import tiktoken

# Get the tokenizer used by GPT-4o / GPT-4 class models.
enc = tiktoken.get_encoding("o200k_base")

text = "Tokenization is surprisingly important!"

# encode() -> the list of integer token IDs the model actually sees.
tokens = enc.encode(text)
print(tokens)            # e.g. [2488, 2065, 382, ...]
print(len(tokens))       # the token COUNT — this is what you pay for

# decode() -> turn IDs back into text (proves it's reversible).
print(enc.decode(tokens))   # "Tokenization is surprisingly important!"

# Peek at how a single word splits into pieces:
for tid in enc.encode("Tokenization"):
    print(tid, repr(enc.decode([tid])))   # shows the sub-word bricks
```

Key terms:
- **tokenizer** — the algorithm that maps text ↔ token IDs (common method: **Byte-Pair Encoding /
  BPE**, which merges frequent character pairs into single tokens).
- **token count** — `len(enc.encode(text))`; the unit of billing and of context limits.
- different model families use different tokenizers, so counts vary slightly between providers.

## 🚀 Stage 3 — In Practice / Why It Matters

Cost math: APIs charge per **1 million tokens** (input + output). A 10-page document ≈ 8,000
tokens ≈ ~$0.02 per GPT-4o call; at 100,000 calls/day that's ~$2,000/day. Token counting is how
you budget, and it's the basis of every chunking decision in Phase 2 and context-window management
in Phase 3.

**Most common beginner mistake:** estimating cost or context by counting *words* or *characters*.
Numbers, code, punctuation, and non-English text tokenise differently — sometimes 2–4 tokens for
one "word." The fix: always measure with the real tokenizer (`tiktoken`). The reason: the model
and the bill operate on tokens, full stop.

## 🛑 STOP — Self-Check

Roughly how many words fit in a model with a 128,000-token context window, and why can't you
answer that exactly just by reading the text's character count?

<details><summary>Answer</summary>

Roughly **~96,000 words** (128,000 × 0.75). You can't be exact from characters because tokenisation
is content-dependent — code, numbers, rare words, and other languages split into more tokens per
"word," so only running the actual tokenizer gives a precise count.
</details>
