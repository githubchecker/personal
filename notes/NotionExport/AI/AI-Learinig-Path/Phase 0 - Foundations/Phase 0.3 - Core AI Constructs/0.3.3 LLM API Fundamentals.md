## 🗺️ Stage 0 — Concept Map

This is where everything in Phase 0 converges: actually **talking to an LLM from code**. You'll
send a structured conversation, control the output with a few dials, and read the reply. It uses
tokens (0.3.1) for cost, and it's the doorway to all of Phase 1 (building AI services).

## 🎈 Stage 1 — The Simple Idea (analogy: a structured conversation, not a search box)

A chat LLM doesn't take a single string — it takes a **transcript of a conversation** made of
labelled turns. You provide the turns so far; the model writes the next one. It's like handing an
actor a script with role labels and saying "now deliver the next line."

**The "Aha!":** the model has **no memory** between calls. Each request must include the whole
relevant conversation; "memory" is just you re-sending earlier turns.

## ⚙️ Stage 2 — How It Actually Works

### The `messages` array (three roles)
- **system** — the standing instructions / persona ("You are a concise assistant"). Highest
  authority.
- **user** — what the human said.
- **assistant** — what the model said previously (you include these to give it "memory").

### The output dials
- **temperature** — randomness. `0` = deterministic/repeatable; `0.7` = more creative/varied.
- **top-p** — restrict choices to the smallest set of tokens whose probability sums to *p*
  (e.g. `0.9`). Tune temperature *or* top-p, not both.
- **max_tokens** — a cap on how many tokens the reply may use.

```python
# pip install openai ; set your key as an environment variable OPENAI_API_KEY
from openai import OpenAI

client = OpenAI()   # reads the API key from the environment automatically

response = client.chat.completions.create(
    model="gpt-4o-mini",                  # which model to call
    messages=[                            # the conversation transcript
        {"role": "system", "content": "You are a concise assistant."},
        {"role": "user",   "content": "Name three primary colors."},
    ],
    temperature=0,                        # deterministic answer
    max_tokens=50,                        # cap the reply length
)

# The reply text lives at this path in the response object.
print(response.choices[0].message.content)

# Usage tells you the token cost of this call (what you're billed on).
print(response.usage.total_tokens)
```

### Streaming (awareness)
With `stream=True`, the reply arrives token-by-token (like ChatGPT typing), letting you show
output as it's generated instead of waiting for the whole thing. You'll build this in Phase 1.

## 🚀 Stage 3 — In Practice / Why It Matters

This single call shape — `messages` + dials → `choices[0].message.content` — is the foundation of
every LLM feature you'll build: chatbots, RAG answerers, agents. The same pattern works across
providers (OpenAI, Anthropic, Azure OpenAI) with minor differences.

**Most common beginner mistakes:**
1. **Hardcoding the API key** in code. Keep it in an environment variable (or a `.env` file loaded
   with `python-dotenv`) — never commit secrets. Reason: leaked keys = real money stolen.
2. **Expecting the model to remember** a previous call. It won't; resend the relevant turns.
3. Using a high temperature for tasks that need consistency (classification, routing). Use `0`.

## 🛑 STOP — Self-Check

You're building a feature that classifies support tickets into fixed categories. What
**temperature** do you choose and why — and where do the previous messages need to go for the
model to use conversation history?

<details><summary>Answer</summary>

Use **temperature `0`** — classification needs deterministic, repeatable output, not creativity.
Conversation history goes into the **`messages` array** as prior `user`/`assistant` turns; the
model is stateless, so anything it should "remember" must be included in that array on every call.
</details>
