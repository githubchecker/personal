## 🎯 Goal

Build a tool that splits a long document into **chunks that each stay under a token budget**,
tracking the token count per chunk so nothing overflows a model's context window. This is the
exact preprocessing step every RAG pipeline (Phase 2) begins with.

## 📄 `splitter.py` (fully commented)

```python
import tiktoken

# The tokenizer that decides how text becomes tokens (and thus the count we budget against).
enc = tiktoken.get_encoding("o200k_base")

def count_tokens(text: str) -> int:
    """Return how many tokens `text` uses for this model family."""
    return len(enc.encode(text))

def split_by_tokens(text: str, max_tokens: int = 100) -> list[str]:
    """
    Split `text` into chunks, each <= max_tokens.
    Strategy: build chunks word-by-word; start a new chunk before exceeding the budget.
    """
    words = text.split()          # naive split on whitespace into words
    chunks: list[str] = []        # finished chunks go here
    current: list[str] = []       # words for the chunk we're building

    for word in words:
        # What would the chunk look like if we added this word?
        candidate = " ".join(current + [word])
        if count_tokens(candidate) > max_tokens and current:
            # Adding it would overflow -> seal the current chunk and start a new one.
            chunks.append(" ".join(current))
            current = [word]
        else:
            current.append(word)

    if current:                   # don't forget the last, partially-filled chunk
        chunks.append(" ".join(current))
    return chunks

# --- Try it ---
document = ("Retrieval augmented generation lets a model answer questions using your own "
            "documents. First we split the text into chunks. Then we embed each chunk into a "
            "vector. At query time we find the closest chunks and feed them to the model. ") * 5

chunks = split_by_tokens(document, max_tokens=40)

for i, chunk in enumerate(chunks):
    print(f"chunk {i}: {count_tokens(chunk)} tokens")   # each should be <= 40

print(f"\n{len(chunks)} chunks; total tokens "
      f"{sum(count_tokens(c) for c in chunks)}")
```

Every printed chunk should report **≤ 40 tokens**, proving no chunk can overflow the budget.

## 🧠 Why this matters
A model can only "see" a fixed number of tokens at once. If a document is larger, you must split
it. Chunking by **token count** (not characters or paragraphs) guarantees each chunk fits, and
each becomes one searchable unit in Phase 2's RAG pipeline.

## 🛑 Reflection (Phase 0 capstone)
Real RAG splitters add a small **overlap** between consecutive chunks (e.g. repeat the last ~10
tokens at the start of the next chunk). Why would overlapping chunks improve retrieval quality?

<details><summary>Answer</summary>

Overlap prevents losing meaning at chunk **boundaries**. A sentence or idea split across two
chunks might otherwise be half-represented in each; repeating a little context at the seam means a
relevant idea stays whole in at least one chunk, so the right text is more likely to be retrieved.
</details>
