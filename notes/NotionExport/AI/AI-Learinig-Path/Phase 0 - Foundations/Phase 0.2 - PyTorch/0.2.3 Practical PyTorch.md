## 🗺️ Stage 0 — Concept Map

This closes Module 0.2 with the *engineering* side of PyTorch: how big a model is, how to run it
efficiently, and — crucially — **when not to use PyTorch at all**. It builds on tensors (0.2.1)
and Hugging Face (0.2.2) and sets up smart cost/latency decisions you'll make throughout the
Road Map.

## 🎈 Stage 1 — The Simple Idea (analogy: knowing a truck's weight before you drive it)

Before moving cargo you check the truck's weight and whether the road can take it. Same with
models: before running one you estimate its **memory footprint** and pick the right "road" (CPU,
GPU, or a managed API). An AI engineer's job is largely *choosing and sizing* models, not building
them.

**The "Aha!":** model size is just arithmetic — *parameters × bytes-per-number*. Once you can do
that sum, you can predict whether a model fits on your hardware.

## ⚙️ Stage 2 — How It Actually Works

### Sizing a model (the key formula)
`memory ≈ number_of_parameters × bytes_per_parameter`

Bytes per parameter depend on **precision (dtype)**:
- `float32` = 4 bytes (full precision)
- `float16` / `bfloat16` = 2 bytes (half precision — common for inference)
- 4-bit quantised ≈ 0.5 bytes (**quantisation** = storing weights in fewer bits to shrink/speed)

```python
# A 7-billion-parameter model:
params = 7_000_000_000
print(params * 4 / 1e9, "GB at float32")   # ~28 GB  (needs a big GPU)
print(params * 2 / 1e9, "GB at float16")   # ~14 GB
print(params * 0.5 / 1e9, "GB at 4-bit")   # ~3.5 GB (fits a laptop)
```

This is why the same "7B" model can need 28 GB or 3.5 GB depending on precision.

### Running efficiently
```python
import torch

# inference_mode() tells PyTorch "we're only USING the model, not training" ->
# it skips bookkeeping needed for learning, saving memory and time.
with torch.inference_mode():
    ...  # run the model here
```

- **`torch.inference_mode()`** (or `model.eval()`) — use whenever you're not training.
- **device choice** — `cuda` (GPU) for large models, `cpu` for small ones.

### When NOT to use PyTorch
If a managed API (OpenAI, Azure OpenAI) already serves the model you need, calling that API is
often the right answer — no GPUs to manage, no weights to host. Reach for local PyTorch when you
need privacy, offline use, or cost control at high volume.

## 🚀 Stage 3 — In Practice / Why It Matters

In Phase 1 you'll choose between a hosted API and a local model; in Phase 4 you'll route simple
tasks to small cheap models and hard tasks to big ones. All of that rests on the sizing formula
and knowing the precision trade-off.

**Most common beginner mistake:** trying to load a 70B model at full precision on a small GPU and
hitting "out of memory." The fix: estimate memory first and use a quantised version (or a managed
API). The reason: weights must fit in (V)RAM to run.
