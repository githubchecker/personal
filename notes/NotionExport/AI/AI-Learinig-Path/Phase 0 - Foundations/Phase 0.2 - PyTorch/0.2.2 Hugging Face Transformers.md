## рЯЧЇпЄП Stage 0 вАФ Concept Map

You won't *build* models from scratch вАФ you'll *download and use* ready-made ones. **Hugging
Face** is the "app store" for AI models, and its `transformers` library is how you load and run
them in a few lines. This builds on tensors (0.2.1, since models input/output tensors) and leads
directly to embeddings (0.3.2) and the RAG pipeline in Phase 2.

## рЯОИ Stage 1 вАФ The Simple Idea (analogy: an app store for models)

**Hugging Face Hub** is a giant public library of pre-trained models, each with a name like
`BAAI/bge-small-en-v1.5`. The `transformers` (and `sentence-transformers`) libraries are the
"installer" вАФ give them a model name and they fetch the weights and set everything up.

**The "Aha!":** using a state-of-the-art model is mostly *downloading a file and calling a
function*. The hard, expensive training was already done by big labs (see Phase 0.0 lifecycle).

### Optional aside вАФ neural network intuition (`[OPTIONAL]`)
A model is layers of simple math (multiply by weights, add, apply a squashing function), repeated
many times. **Training** nudges those weights to reduce error (`loss.backward()` computes "which
way to nudge"). For AI engineering you can treat the trained model as a black box вАФ you *use* it,
you don't train it. Watch 3Blue1Brown's "Neural Networks" series once for the visual; no coding
needed.

## вЪЩпЄП Stage 2 вАФ How It Actually Works

Install: `pip install sentence-transformers` (wraps `transformers` for easy embeddings).

```python
from sentence_transformers import SentenceTransformer

# 1. Load a small embedding model by its Hub name (downloads once, then caches locally).
model = SentenceTransformer("BAAI/bge-small-en-v1.5")

# 2. Turn text into embedding vectors (tensors of floats capturing meaning).
sentences = ["A dog runs in the park", "A puppy plays outside"]
embeddings = model.encode(sentences)   # returns a NumPy array, shape (2, 384)

print(embeddings.shape)   # (2, 384) -> 2 sentences, each a 384-number vector
```

Lower-level pieces you'll also meet:
- **`AutoTokenizer`** вАФ turns text into token IDs the model understands (tokens = Phase 0.3.1).
- **`AutoModel`** вАФ loads the model weights themselves.
- **`pipeline(...)`** вАФ a one-liner for common tasks (sentiment, summarisation).
- **`model.eval()`** vs **`model.train()`** вАФ *eval* mode = using the model (inference, weights
  frozen); *train* mode = adjusting weights. For us it's almost always `eval`.

## рЯЪА Stage 3 вАФ In Practice / Why It Matters

In Phase 2 you'll embed every document chunk with exactly this `model.encode(...)` call, store the
vectors, and later compare them to a question's vector to find relevant text. Hugging Face is also
where you'll grab open-weight chat models (Llama, Mistral, Phi) for local use.

**Most common beginner mistake:** using a *different* embedding model for documents than for
queries. The vectors then live in different "meaning spaces" and retrieval quality collapses. The
fix: use the **same** model for both. The reason: similarity only makes sense within one model's
coordinate system.

and why does the model name matter for later retrieval?

<details><summary>Answer</summary>

Shape **`(100, 384)`** вАФ 100 rows (one per sentence), each a 384-number vector. The model name
matters because embeddings are only comparable when produced by the **same** model; switching
models for queries vs documents puts them in incompatible vector spaces and breaks search.
</details>
