# 0.0.3 — Encoder vs Decoder (the split that matters for RAG)

> Phase 0.0 · Module 3 of 6 · `[MUST KNOW]` · reading-only

## 🗺️ Stage 0 — Concept Map

This is the **most practically important** concept module: it decides **which model you use at each
stage of a RAG pipeline**. It builds directly on the BERT-vs-GPT "family split" from the last
lesson, and it's the foundation of everything in Phase 2.

## 🔑 New Terms (plain English)

- **Encoder model** — reads text and turns it into a **meaning-vector (embedding)**. Understands; doesn't write.
- **Decoder model** — **generates** text one token at a time (predicts the next word). The "LLM" you chat with.
- **Encoder-decoder** — reads an input and writes a transformed output (translation, summarisation).
- **Embedding** — the vector an encoder produces (meaning expressed as numbers).
- **Autoregressive** — generating one token, then feeding it back to produce the next, and so on.

## 🎈 Stage 1 — The Simple Idea (analogy: a librarian vs. a storyteller)

An **encoder** is a **librarian**: it reads a passage and files it by *meaning* so you can find it
later — but it never writes new stories. A **decoder** is a **storyteller**: it writes the next
word, then the next, building text — but it isn't built to file things by meaning. **RAG needs
both:** the librarian to find the right passage, the storyteller to answer using it.

**The "Aha!":** "understanding" and "generating" are **two different model families**. A RAG system
uses one of each.

## ⚙️ Stage 2 — How It Actually Works

- **Encoder (the "understanding" family):** reads text → a dense **embedding** capturing meaning.
  *Bidirectional* (sees all words in both directions at once). Examples: BERT, `text-embedding-3`,
  `bge-large`, `nomic-embed`. **Cannot generate text.** In RAG, an encoder embeds every document chunk.
- **Decoder (the "generation" family):** takes tokens and predicts the **next** token, one at a time
  (**autoregressive**). *Unidirectional* (each token sees only the ones before it). Examples: GPT-4o,
  Claude, Llama, Mistral. **Every LLM chat call is a decoder.** The loop `token1 → token2 → … → END`
  is why **streaming** exists.
- **Encoder-decoder (the "translation" family):** T5, BART — read input, write output. You won't
  build with these directly; just know they exist.
- **The RAG mapping (the key takeaway):**
  - **Ingestion time:** encoder → document chunks → embeddings → store in a vector database.
  - **Query time:** the **same** encoder → user question → query vector → retrieve nearest chunks →
    **decoder (LLM)** → reads those chunks → writes the answer.
  - **Critical rule:** use the **same encoder** for ingestion and query, or the vectors won't align
    and retrieval quality collapses.

### 📊 The two-model RAG flow
```
  Documents ─►[ Encoder ]─► vectors stored in a vector DB
  Question  ─►[ Encoder ]─► query vector ─► nearest chunks ─►[ Decoder / LLM ]─► Answer
```

## 🚀 Stage 3 — In Practice / Why It Matters

This is the backbone of every RAG system in Phase 2 — at minimum **one encoder + one decoder**
working together. Knowing which is which stops a whole class of mistakes, like trying to "generate"
an answer with an embedding model, or comparing vectors made by two different encoders.

## 🧠 Common Misconceptions

- **"One model does everything in RAG."** → Usually **two**: an encoder (embeddings) and a decoder
  (the answer).
- **"Embedding models can answer questions."** → No — encoders only produce vectors; they can't
  generate text.
- **"Any embedding model works at query time."** → No — you must use the **same** model that
  embedded the documents, or the vector spaces don't line up.

## 📌 Quick Reference (key facts)

- **encoder = understand → embeddings** (BERT, bge, text-embedding-3). Bidirectional.
- **decoder = generate → text** (GPT-4o, Claude, Llama). Unidirectional, autoregressive.
- **RAG:** encoder to ingest + **same** encoder to query + decoder to answer.
- **streaming** exists because decoders emit one token at a time.

> 🎯 **Interview angle:** "Which family produces embeddings, and why can't it generate?" → the
> **encoder**; it's built to read and represent meaning, not to predict next tokens.

## 🛑 STOP — Self-Check

Which model family do you use for **RAG embeddings**, and **why can't it generate text**?

<details><summary>Answer</summary>

You use an **encoder** model. An encoder is *bidirectional* and trained to **represent** text as a
meaning-vector (embedding) — it reads and compresses meaning, but it has no mechanism to **predict
the next token**, which is what text generation requires. Generating answers is the job of a
**decoder** (the LLM). A RAG pipeline uses an encoder to find the right text and a decoder to write
the answer.
</details>
