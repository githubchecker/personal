# 0.0.5 — Core Vocabulary (tokens, context window, temperature, hallucination…)

> Phase 0.0 · Module 5 of 6 · `[MUST KNOW]` · reading-only

## 🗺️ Stage 0 — Concept Map

These are the words in **every** job description, architecture discussion, and vendor call. They
need to be in your working vocabulary *before* you write any code. This pulls together ideas you'll
use constantly in Phases 1–4.

## 🔑 New Terms (plain English)

- **Token** — a chunk of text the model reads (≈ ¾ of a word).
- **Context window** — the most tokens a model can consider at once.
- **Temperature** — a "randomness" dial for the output (`0` = always the safest choice).
- **Top-P / Top-K** — alternative dials that limit which next-words are considered.
- **System / User / Assistant** — the three roles in a chat conversation.
- **Hallucination** — a confident, made-up answer. **Grounding** — basing answers on real sources.

## 🎈 Stage 1 — The Simple Idea (analogy: a whiteboard for working memory)

The **context window** is a fixed-size **whiteboard**. Everything the model can "see" right now —
your instructions, the conversation so far, any retrieved documents, and its own answer — must fit
on that whiteboard, measured in **tokens**. Run out of room and the oldest notes get wiped.
**Temperature** is how adventurous its handwriting is: `0` always writes the single most likely next
word; higher values let it wander into more creative choices.

**The "Aha!":** the model has **no long-term memory** between calls — only what's on the whiteboard
(the context window) *this* time. Any "memory" is just you re-writing earlier notes back onto it.

## ⚙️ Stage 2 — How It Actually Works

- **Tokens:** models read tokens, not words. Common word = 1 token; rare/long words = 2–4. **~1,000
  tokens ≈ 750 words ≈ 1.5 pages.** You're billed per token (input + output) — this drives chunking
  (Phase 2) and cost (Phase 4).
- **Context window:** `input + output tokens ≤ window`. Current sizes: GPT-4o ≈ 128K, Claude ≈ 200K,
  Gemini ≈ 1M. The **"lost in the middle"** problem: models do worse on info buried in the middle of
  a long context — a key reason **chunk + retrieve (RAG)** often beats stuffing everything in.
- **Temperature:** `0` = deterministic/repeatable (use for routing & classification); `0.3–0.7` =
  balanced; `0.7–1.0` = creative. **Agent rule:** temperature `0` on routing/supervisor steps;
  higher only on a final creative-writing step.
- **Top-P (nucleus):** consider only the smallest set of words whose probabilities add up to `P`
  (e.g., `0.9`). **Top-K:** consider only the `K` most likely words. Tune temperature **or** top-p,
  not both.

**🔬 How the model actually picks the next word (the mechanism behind those dials):**
1. The transformer's real output isn't a word — it's **logits**: a raw score for **every** word in
   the vocabulary (~50k–100k scores).
2. **Temperature** is applied **first**, scaling those scores — low temp **sharpens** the gap (the
   top word dominates → near-deterministic); high temp **flattens** it (more words get a real chance).
3. **Softmax** turns the scaled scores into **probabilities** that sum to 100%.
4. **Top-K / Top-P** then **filter** the candidates; the model **samples** one token from what's
   left, then loops (autoregressive — see [0.0.7](0.0.7%20How%20a%20Transformer%20Works.md)).

So `temperature` / `top_p` in an API call simply reshape this score → probability step; temperature
`0` skips the randomness and always takes the top word. *(A common interview question.)*

- **Roles:** **system** = standing instructions/persona (highest authority); **user** = the human;
  **assistant** = the model's prior replies — feeding these back is how an agent "remembers."
- **Hallucination:** the model predicts the *statistically likely* next token — it doesn't look up
  facts. Where training data is thin, it fills the gap with a **plausible fabrication, stated
  confidently**. **RAG** mitigates this by putting real source text on the whiteboard (**grounding**).
- **Three kinds of memory:** **parametric** (baked into the weights, static), **in-context** (on the
  whiteboard now — RAG uses this), **external** (databases/vector stores — agents use this).

### 📊 The budget that governs every call
```
  system prompt  +  conversation history  +  retrieved chunks  +  output   ≤   context window
```

## 🚀 Stage 3 — In Practice / Why It Matters

Token counting drives every chunking and cost decision; temperature settings appear in every agent;
the three roles map straight to `SystemMessage` / `HumanMessage` / `AIMessage` in LangChain
(Phases 1–3). Grounding via RAG is how you fight hallucination, and evaluation (Phase 4) measures it.

## 🧠 Common Misconceptions

- **"The model remembers our past chats."** → No — it's **stateless**; you must resend the relevant
  turns every call.
- **"A bigger context window is always better."** → Big windows still suffer "lost in the middle";
  targeted retrieval often beats stuffing everything in.
- **"Higher temperature = smarter."** → No — it's more *random/creative*, often **less** accurate.
  Use `0` when you need consistency.
- **"Hallucination means the model is broken."** → It's inherent to next-token prediction; grounding
  reduces it, it isn't a bug to be "fixed."

## 📌 Quick Reference (key facts)

- **1,000 tokens ≈ 750 words ≈ ~1.5 pages.**
- **how a token is chosen:** logits → (temperature scales) → softmax → (top-k/top-p filter) → sample.
- **temperature 0** = deterministic (routing/classification); higher = creative (final output).
- tune **temperature OR top-p**, not both.
- roles: **system** (authority) / **user** / **assistant** (memory via resending).
- **RAG grounds answers → fewer hallucinations.**

> 🎯 **Interview angle:** "Settings for an agent's routing node?" → temperature `0`, top-p `1` for
> fully deterministic routing; raise them only on a final creative step.

## 🛑 STOP — Self-Check

**Why does hallucination happen, and how does RAG reduce it?**

<details><summary>Answer</summary>

Hallucination happens because an LLM is trained to predict the **statistically most likely next
token**, not to retrieve verified facts. When its training data on a topic is sparse, it still
produces a fluent, **plausible-but-fabricated** answer — stated confidently. **RAG** reduces it by
**retrieving real source text and putting it in the context window** ("grounding"), so the model
answers *from* provided evidence instead of guessing from memory.
</details>
