# 0.0.4 — The Model Lifecycle (who does what — and what *your* job is)

> Phase 0.0 · Module 4 of 6 · `[MUST KNOW]` · reading-only

## 🗺️ Stage 0 — Concept Map

This answers two standard interview questions at once: **"What is your job as an AI engineer?"** and
**"Why RAG instead of fine-tuning?"** It maps the five stages a model passes through — from a
$50M pre-training run to the application code *you* write — and shows where your work actually sits.

## 🔑 New Terms (plain English)

- **Pre-training** — the big labs train a raw "base" model on trillions of words (costs $50M+).
- **SFT (Supervised Fine-Tuning)** — teach that base model to follow instructions.
- **RLHF / DPO** — align it to be safe and helpful using human preferences.
- **Fine-tuning / LoRA** — cheaply nudge a finished model toward your domain's *style/behaviour*.
- **RAG / Agents** — a **frozen** model plus *your* code connecting it to data and tools (your main job).

## 🎈 Stage 1 — The Simple Idea (analogy: building vs. furnishing a house)

The big labs **build the house** (pre-training) and do the major fit-out (SFT, RLHF) — spending
tens of millions. You don't rebuild the house. **Your job is furnishing and wiring it for a
purpose:** connecting the finished model to *your* data (RAG) and *your* tools (agents). Once in a
while you repaint a room (a small fine-tune). Around **95% of enterprise AI work is the furnishing**,
not the construction.

**The "Aha!":** as an AI engineer/architect you **almost never touch the model's weights** — you
build the **application layer** around a frozen model.

## ⚙️ Stage 2 — How It Actually Works (the 5 stages)

| Stage | What happens to the weights | Who does it | Your role |
| --- | --- | --- | --- |
| **1. Pre-training** | ALL parameters trained on trillions of tokens | Meta/OpenAI/Google ($50M+) | none |
| **2. SFT** | ALL parameters tuned on (question, ideal answer) pairs | the same labs | none |
| **3. RLHF / DPO** | ALL parameters tuned on human preference rankings | the same labs | none (awareness) |
| **4. Fine-tuning / LoRA** | a **tiny** adapter trained; originals frozen | ML/AI engineers — *you, occasionally* | sometimes (Phase 4) |
| **5. RAG / Agents** | **ZERO** weights change | AI engineers — **you** | **primary** |

- **Stage 1–3** produce the public "chat" model (e.g., the difference between Llama "base" and
  "instruct" is SFT+RLHF on the *same* weights).
- **Stage 4 — LoRA:** freezes the original weights and trains tiny add-on matrices (≈1/1000th the
  cost). Use it for **style/tone/format** — **not** to inject facts.
- **Stage 5 — RAG & Agents:** weights are **frozen**. You write app code: retrieve documents → put
  them in the prompt (RAG); or let the model call tools in a loop (agents). Cost: cents per query.
- **Fine-tune vs RAG (the decision):** fine-tuning changes **behaviour/style**; RAG adds
  **knowledge/facts**. "The model doesn't know our policies" → that's a **RAG** problem.
- **Cost ladder:** pre-training ($50M+) → fine-tuning ($500–$5,000) → RAG (cents/query).

## 🚀 Stage 3 — In Practice / Why It Matters

This frames your **entire career**: the retrieval system, the orchestration logic, the evaluation
pipeline, the cost governance — the application layer around frozen models. It's why **RAG is the
default** and fine-tuning is rarely the first answer.

## 🧠 Common Misconceptions

- **"To make the model know our data, we must fine-tune it."** → Usually **RAG** is right;
  fine-tuning doesn't reliably add facts — it changes behaviour.
- **"AI engineers mostly fine-tune models."** → No — mostly **RAG + agents**, with weights frozen.
- **"Teaching it in the prompt permanently trains it."** → No — that's **in-context** only,
  forgotten the moment the call ends.

## 📌 Quick Reference (key facts)

- pipeline: **pre-train → SFT → RLHF/DPO** (labs, all weights) → **LoRA** (you, tiny) → **RAG/agents** (you, no weights).
- **fine-tune = behaviour/style · RAG = knowledge/facts.**
- **your job = the application layer around a frozen model.**
- cost ladder: **$50M+ → $500–$5k → cents/query.**

> 🎯 **Interview angle:** "Why RAG instead of fine-tuning to add company knowledge?" → RAG injects
> facts at query time — cheaper, instantly updatable, and traceable to a source; fine-tuning changes
> behaviour and is unreliable for facts.

## 🛑 STOP — Self-Check

A client wants the model to answer questions from their **internal policy documents**. Do you
**fine-tune** or use **RAG** — and why?

<details><summary>Answer</summary>

Use **RAG**. The need is **factual knowledge** ("what do our policies say?"), and RAG injects the
actual document text into the prompt at query time — cheap, instantly updatable when policies
change, and the answer can be traced back to a source. **Fine-tuning** changes a model's
*behaviour/style*, doesn't reliably implant new facts, and would need redoing every time the docs
change. Fine-tuning before having a working RAG system is the wrong order of operations.
</details>
