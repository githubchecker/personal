# 0.0.1 — What Is an AI Model? (Parameters & Weights)

> Phase 0.0 · Module 1 of 6 · `[MUST KNOW]` · reading-only (no code)

## 🗺️ Stage 0 — Concept Map

This is the **first idea in all of AI**, and almost every later decision traces back to it: how much
a model costs, how fast it runs, whether you can run it on a laptop, and why we use RAG instead of
retraining. Master this and the rest of the roadmap stops feeling like magic. Nothing comes before
it; it unlocks model sizing, cost-routing, and the open-vs-closed choice.

## 🔑 New Terms (plain English)

- **Model** — a program that "learned" patterns; really just a big **file of numbers** plus math to use them.
- **Parameter / weight** — one of those numbers (e.g., `0.347`). Models have millions or billions.
- **Inference** — *using* a trained model to get an output (running it). Training is making it; inference is using it.
- **Quantisation** — storing each number in fewer bits to shrink the file and speed it up.
- **Open-weight vs closed** — downloadable & self-hostable, vs API-only (the numbers are kept secret).

*(Anything unfamiliar? The [AI Terms glossary](../../AI%20Terms%20-%20Plain%20English%20Glossary.md) has one-line definitions.)*

## 🎈 Stage 1 — The Simple Idea (analogy: a mixing board with millions of knobs)

Picture a giant audio mixing board covered in **billions of knobs**. Each knob is a **parameter**.
Before training, every knob is set randomly, so the model outputs gibberish. **Training** is the
slow process of twisting every knob until the output "sounds right." When training stops, the knob
positions are frozen — and *that frozen set of knob positions is the model.*

**The "Aha!":** an AI model is literally a **big file of numbers** (the knob settings). Its entire
"knowledge" — grammar, facts, reasoning patterns — is encoded in those numbers. Nothing else.

## ⚙️ Stage 2 — How It Actually Works

- A model is a **mathematical function saved as a file of numbers**. Each parameter is a decimal
  number. Download Llama 3.1 8B and you're downloading ~8 billion such numbers.
- **Size tiers** (and what each is for):
  - **7–8B** (Llama 3 8B, Phi-4, Mistral 7B): fast, cheap, runs on a laptop GPU. Good for simple
    Q&A, classification, routing.
  - **70B** (Llama 3.1 70B, Qwen2.5 72B): strong reasoning, follows complex instructions. Needs a
    server/cloud GPU. The enterprise workhorse.
  - **400B+** (GPT-4o, Claude, Gemini, Llama 405B): frontier reasoning, multimodal. API-only for most teams.
- **Why bigger = smarter but slower & pricier:** 10× more knobs ≈ 10× more computation for every
  word generated.
- **The one formula — memory ≈ parameters × bytes-per-number:**
  - `float32` = 4 bytes, `float16` = 2 bytes, **4-bit quantised ≈ 0.5 bytes**.
  - A 7B model: ~14 GB at float16, but only **~4 GB at 4-bit** — small enough for a laptop.
- **Formats:** `.safetensors` (the Hugging Face standard), `GGUF` (for running locally with tools like Ollama).
- **Closed vs open-weight:**
  - *Closed* (GPT-4o, Claude, Gemini): API only, weights hidden, your data leaves your machine on every call.
  - *Open-weight* (Llama, Mistral, Phi, Qwen): downloadable, self-hostable, **your data stays put**.

### 📊 The mental picture
```
  input text  ──►  [  a function with billions of parameters  ]  ──►  next word
                       (the model = this box of frozen numbers)
```

## 🚀 Stage 3 — In Practice / Why It Matters

- **Cost-routing** — send easy tasks to a small 7B model and hard ones to a 70B/frontier model — is
  a core skill you'll build in Phase 4. It only makes sense once you understand parameters = cost.
- **Why enterprises choose open-weight:** healthcare (HIPAA), finance, legal, government — anywhere
  data *cannot* leave their infrastructure. Open-weight models run in-house.
- Model choice is therefore an **architecture decision**, not a minor detail.

## 🧠 Common Misconceptions

- **"The model looks facts up in a database."** → No. It *computes* the next word from its
  parameters. That's exactly why it can confidently make things up (hallucinate).
- **"A bigger model is always better."** → No — bigger means slower and costlier; many tasks run
  fine on a 7B.
- **"Downloading a model downloads its training data."** → No. You only get the numbers (weights),
  never the data it learned from.
- **"Chatting with it changes the model."** → No. At inference the weights are **frozen**; it
  doesn't remember you after the chat unless *you* store that separately.

## 📌 Quick Reference (key facts)

- **parameter = one number** (a "knob"). **model = a file of billions of them.**
- **memory ≈ params × bytes/param** (fp32 = 4, fp16 = 2, 4-bit ≈ 0.5).
- tiers: **7B** laptop · **70B** server · **400B+** API-only.
- **closed** = API, data leaves · **open-weight** = self-host, data stays.

> 🎯 **Interview angle:** "Why is model size a cost/latency lever?" → more parameters = more compute
> per token = higher cost and latency, so route simple tasks to smaller models.

## 🛑 STOP — Self-Check

In your own words: **why does a 70B model cost more to run than a 7B model?**

<details><summary>Answer</summary>

A 70B model has roughly **ten times as many parameters** (numbers) as a 7B model, and the model
must do a calculation across those parameters to produce **each token**. Ten times more numbers ≈
ten times more computation per word, which means more expensive hardware, more memory, and higher
latency. That's why we route simple tasks to small models and reserve big ones for hard tasks.
</details>
