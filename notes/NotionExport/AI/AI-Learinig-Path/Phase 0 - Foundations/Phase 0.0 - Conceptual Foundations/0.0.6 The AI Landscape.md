# 0.0.6 — The AI Landscape (types of models, when to use each)

> Phase 0.0 · Module 6 of 6 · `[MUST KNOW]` · reading-only

## 🗺️ Stage 0 — Concept Map

The final concept module: a **map of model types** so you never recommend the wrong *class* of tool
in an architecture review. It ties the vocabulary you just learned to real decisions, and it's the
last thing you need before reading any AI engineering blog, JD, or design doc without getting lost.

## 🔑 New Terms (plain English)

- **Discriminative AI** — **labels/classifies** existing data (spam? fraud? which category?).
- **Generative AI** — **creates** new content (text, images, code).
- **Foundation model** — large, general, adaptable to many tasks (GPT-4o, Llama).
- **Task-specific model** — fine-tuned for **one** narrow job (fast, cheap, narrow).
- **Multimodal** — handles more than text (images, sometimes audio/video).
- **Reasoning model** — "thinks" step-by-step before answering (slower, pricier, better at hard problems).

## 🎈 Stage 1 — The Simple Idea (analogy: a hospital's specialists vs. generalists)

A **foundation model** is a brilliant **generalist doctor** — handles almost anything, but costs
more per visit. A **task-specific model** is a **specialist** who does one procedure faster and
cheaper but nothing else. **Classic ML** (discriminative) is the **lab test** — perfect for a
specific yes/no question (fraud? spam?) and far cheaper than asking the generalist. A good architect
sends each case to the right one.

**The "Aha!":** "use an LLM" isn't always the answer — you **match the model class to the job's
cost, latency, and accuracy needs**.

## ⚙️ Stage 2 — How It Actually Works (the choices)

- **Discriminative vs Generative:** discriminative *labels* input → output (spam, fraud,
  classification); generative *creates* content. **Where classic ML still wins:** anomaly detection,
  fraud scoring, recommendations, forecasting — a small XGBoost model beats an LLM on cost, latency,
  and accuracy for those.
- **Foundation vs Task-specific:** foundation = broad, varied tasks; task-specific = high-volume,
  latency-sensitive, narrow jobs (classify 10,000 tickets/day). Faster and cheaper for that one job,
  useless for anything else.
- **Closed vs Open-weight (recap):** closed (GPT-4o, Claude, Gemini) = API, best quality, data
  leaves; open-weight (Llama, Mistral, Phi, Qwen) = self-host, full data control.
- **Multimodal:** GPT-4o, Claude, Gemini handle text **+ images** (some add audio/video). Used to
  read document-page images and charts (Phase 2 multimodal RAG). You don't need to *train* these.
- **Reasoning vs standard chat:** standard (GPT-4o) = one quick pass (1–5s), good for most tasks;
  **reasoning** (o3, DeepSeek-R1, Claude extended thinking) = internal step-by-step thinking, slow
  (10–120s) and **5–20× the cost** — use only where standard models demonstrably fail (multi-hop
  analysis, hard code review). Routing these is a Phase 4 skill.

### 📊 A quick decision sketch
```
  simple / narrow / high-volume   ─►  small or task-specific model (or classic ML)
  broad / varied                  ─►  foundation model
  needs hard multi-step reasoning ─►  reasoning model (only when standard fails)
  involves images / charts        ─►  multimodal model
```

## 🚀 Stage 3 — In Practice / Why It Matters

Architects pick the **class** first, then the specific model. **Cost-routing** across classes —
cheap model for simple work, frontier/reasoning for the hard 5% — is core LLMOps (Phase 4).
Recommending a 400B reasoning model to classify support tickets is exactly the kind of expensive
mistake this module prevents.

## 🧠 Common Misconceptions

- **"Use an LLM for everything."** → For fraud, anomaly detection, and forecasting, **classic ML**
  is cheaper *and* more accurate.
- **"Reasoning models are just better LLMs — use them always."** → They cost 5–20× more and are far
  slower; reserve them for genuinely hard problems.
- **"Foundation models are always the right pick."** → For one narrow, high-volume task, a
  **task-specific** or small model wins on cost and latency.

## 📌 Quick Reference (key facts)

- **discriminative (label) vs generative (create).**
- **classic ML wins:** fraud, anomaly, recommendations, forecasting.
- **foundation = broad · task-specific = narrow/cheap/fast.**
- **reasoning model** only when standard models fail (5–20× cost).
- **multimodal** = text + images (document/chart understanding).

> 🎯 **Interview angle:** "10,000 support tickets/day to classify — which model?" → a small/
> task-specific model (or classic ML) for cost and latency, **not** a frontier LLM.

## 🛑 STOP — Self-Check

You must classify **10,000 support tickets/day** into 5 fixed categories. Which **model class** do
you choose, and why?

<details><summary>Answer</summary>

A **small / task-specific model** (or even a classic **discriminative ML** classifier). The job is
narrow, high-volume, and latency-sensitive, so you want something **fast and cheap** that does this
one thing well. A frontier or reasoning LLM would be far more expensive and slower per ticket for no
accuracy gain on such a constrained task. (If you do use an LLM, pick a small one and set
temperature `0` for consistent labels.)
</details>
