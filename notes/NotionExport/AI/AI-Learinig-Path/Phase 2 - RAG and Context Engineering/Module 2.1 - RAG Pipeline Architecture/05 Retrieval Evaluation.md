# 05 — Retrieval Evaluation

> Phase 2 · Module 2.1 · Lesson 5 · `[SHOULD — architect-bonus; eval-driven RAG]`

## 🗺️ Stage 0 — Concept Map

**The problem first.** You change the chunk size, or swap `3-small` for `3-large`, and the answers
*feel* better. Did they actually improve, or are you fooling yourself on three cherry-picked questions?
"It feels better" is not engineering. **Retrieval evaluation** replaces vibes with **numbers**: did the
*right chunks* come back in the top-k?

Crucially, RAG has two failure points — **retrieval** (wrong chunks fetched) and **generation** (right
chunks, bad answer). If the answer is wrong because the relevant chunk was never retrieved, no prompt
tweak will fix it. So you **measure retrieval first**.

**Where it sits.** It's the feedback loop around all of Module 2.1: every chunking/embedding choice
(03, 04) and every Module 2.3 upgrade (hybrid, reranking) is *justified by a metric*, not a hunch.

**Why care.** Eval-driven development is what distinguishes an AI *engineer* from someone pasting
prompts. The Lesson 06 milestone literally measures **Recall@5 before and after** a chunking change.

## 🔑 New Terms (plain English)

- **Ground truth** — the known-correct answer key: for each test query, which chunk(s) are relevant.
- **Relevant chunk** — a chunk that actually contains the answer to a query.
- **Recall@K** — of all the relevant chunks, how many appeared in the top-k retrieved.
- **Precision@K** — of the top-k retrieved, what fraction were relevant.
- **Hit Rate** — did *at least one* relevant chunk appear in the top-k? (yes/no, averaged).
- **MRR (Mean Reciprocal Rank)** — how *high* the first relevant chunk ranked (1st = 1.0, 2nd = 0.5…).
- **NDCG** — ranking quality with *graded* relevance, discounted by position.
- **Synthetic QA dataset** — test (question → relevant-chunk) pairs generated by an LLM.
- **Regression test** — re-running the eval set on every change to catch quality drops.
  (See the [glossary](../../AI%20Terms%20-%20Plain%20English%20Glossary.md).)

## 🎈 Stage 1 — The Simple Idea (analogy: a fishing-net test)

You're testing fishing nets. You drop a net (run a query) into a pond where you *know* which fish are
the right ones (ground truth). **Recall@K** = of the right fish in the pond, how many did the net pull
up? **Precision@K** = of everything in the net, how much was the right fish vs junk? **MRR** = was the
right fish near the top of the haul or buried at the bottom? Test many nets the same way and you can
say, with numbers, which net is best.

**The "Aha!":** you can only improve what you measure — and you must measure **retrieval** separately
from the final answer.

## ⚙️ Stage 2 — How It Actually Works

**💢 The old/painful way** — ask the system three questions, read the answers, declare "looks good,
ship it." You optimise the *prompt* while the real bug is that the relevant chunk was never retrieved,
and you have no way to know a change made things *worse* for the other 997 questions.

You need (a) a **test set** of `(query, relevant_chunk_ids)` pairs and (b) **metrics** comparing
retrieved ids to relevant ids.

### 5.1 Recall@K — the primary RAG retrieval metric

"Of the relevant chunks, how many did we retrieve in the top-k?" If the answer isn't retrievable,
nothing downstream can fix it — so this is the metric to watch.

```python
def recall_at_k(retrieved_ids, relevant_ids, k):
    top_k = set(retrieved_ids[:k])
    hits = top_k & set(relevant_ids)
    return len(hits) / len(relevant_ids)          # 1.0 = all relevant chunks were in top-k
```

- **Use when:** the default question "is the answer even retrievable?" Track **Recall@5** / **Recall@10**.

### 5.2 Precision@K and Hit Rate — noise and coverage

```python
def precision_at_k(retrieved_ids, relevant_ids, k):
    top_k = retrieved_ids[:k]
    hits = [d for d in top_k if d in set(relevant_ids)]
    return len(hits) / k                          # fraction of the top-k that was relevant

def hit_rate(retrieved_ids, relevant_ids, k):
    return 1.0 if set(retrieved_ids[:k]) & set(relevant_ids) else 0.0  # at least one?
```

- **Use when:** **Precision@K** when irrelevant chunks are *hurting* the LLM (noise/cost); **Hit Rate**
  for a simple "did we get *anything* useful" signal.

### 5.3 MRR — how high the first relevant chunk ranks

```python
def reciprocal_rank(retrieved_ids, relevant_ids):
    for i, d in enumerate(retrieved_ids, start=1):
        if d in set(relevant_ids):
            return 1.0 / i                        # first relevant at rank i -> 1/i
    return 0.0
# MRR = average of reciprocal_rank over all test queries
```

- **Use when:** rank order matters (you feed only the top 1–3 chunks) — MRR rewards putting the right
  chunk **first**.

### 5.4 NDCG — graded, position-discounted ranking quality

When relevance isn't just yes/no (some chunks are *more* relevant) and the order matters, **NDCG**
scores your ranking against the perfect ranking (1.0 = ideal), and counts a relevant chunk for **less
the further down the list** it sits (that's the *position-discounted* part).

- **Use when:** you have graded relevance labels and care about fine-grained ranking quality (common in
  search teams). For most RAG, Recall@K + MRR are enough; reach for NDCG when ranking is the product.

### 5.5 Building a synthetic QA dataset (your ground truth)

You rarely have hand-labelled data — so generate it. For each chunk, ask an LLM to write a question
*that chunk* answers; the pair `(question, chunk_id)` becomes ground truth.

```python
def make_qa_pair(chunk_text, chunk_id):
    q = client.responses.create(
        model="gpt-5.5",
        input=f"Write ONE question answerable ONLY by this text:\n\n{chunk_text}",
    ).output_text
    return {"question": q, "relevant_ids": [chunk_id]}

qa_set = [make_qa_pair(c.text, c.id) for c in sample_chunks]   # ~100–200 pairs is plenty to start
```

- **Inline tip:** sample chunks across the whole corpus, and **spot-check** generated questions — drop
  vague ones. Libraries like **`ragas`** automate this (and add answer-quality metrics) for production.

### 5.6 Regression testing

Freeze the eval set and re-run it on **every** change — new chunking, new embedder, new data load.
A drop in Recall@5 fails the build, exactly like a unit test.

### 5.7 Evaluating the *generation* half (not just retrieval)

Recall@K and MRR measure **retrieval**. But even with perfect chunks, the LLM can still answer badly — so a
complete RAG eval also scores the **answer**. The standard frameworks (**RAGAS**, **DeepEval**) use
**LLM-as-judge** metrics; the four you'll quote:

- **Faithfulness / groundedness** — is *every claim* in the answer supported by the retrieved context?
  (this is the **hallucination** detector — low score = the model made something up).
- **Answer (response) relevancy** — does the answer actually address the question, without waffling?
- **Context precision** — were the retrieved chunks *relevant* (low noise) and well-ranked?
- **Context recall** — did retrieval bring back *all* the context needed to fully answer?

```python
# RAGAS sketch
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall
result = evaluate(dataset, metrics=[faithfulness, answer_relevancy, context_precision, context_recall])
```

> **The two halves, decoupled:** *context precision/recall* grade **retrieval**; *faithfulness/answer-
> relevancy* grade **generation**. This tells you *where* to fix: high context-recall but low **faithfulness**
> means the model is **ignoring good context** — a prompt/generation fix, not a retrieval one. (LLM-judge
> metrics cost model calls and can be noisy, so sample and spot-check, like any judge.)

> 🔬 **Under the hood:** every metric is just **set/rank math on ids**. The only hard part is the
> **ground truth** — once you have `(query → relevant_chunk_ids)`, retrieval becomes a measurable
> classification/ranking problem. That's why synthetic QA generation is the key unlock: it manufactures
> the answer key cheaply so the rest is arithmetic.

## 🚀 Stage 3 — In Practice / Why It Matters

Mature RAG teams run a retrieval eval set in **CI** (the automated build that runs on every change).
The workflow: build a synthetic QA set → measure
**Recall@K** + **MRR** for the current pipeline → change one thing (chunk size, embedder, add reranking)
→ re-measure → keep it only if the numbers rise. This is precisely how you'll justify the Module 2.3
upgrades (hybrid search, reranking) and how the Lesson 06 milestone proves a chunking change helped.

## ⚖️ Variations & When to Use

| Metric | Answers… | Reach for it when… |
|---|---|---|
| **Recall@K** | "Is the answer retrievable at all?" | **always** — the primary RAG metric |
| Precision@K | "How much noise is in the top-k?" | irrelevant chunks hurt the LLM / cost |
| Hit Rate | "Did we get anything useful?" | quick, coarse signal |
| **MRR** | "Is the best chunk near the top?" | you feed only top 1–3 chunks |
| NDCG | "Is the whole ranking well-ordered?" | graded relevance; search-quality work |

> Decision rule: **track Recall@K first (retrievability); add MRR when rank matters; Precision@K when
> noise hurts; NDCG only for graded ranking work.**

## 🐛 Common Misconceptions

| Misconception | Why it's wrong | The right view |
|---|---|---|
| "Tune the prompt to fix bad answers." | Often the chunk was never retrieved. | Measure **retrieval** before generation. |
| "Three test questions are enough." | Too small to be reliable. | ~100–200 queries; track the average. |
| "Recall and precision are the same." | Recall = of relevant, how many found; precision = of found, how many relevant. | They trade off; know which you need. |
| "High Recall@K = good answers." | Generation can still mangle good context. | Also evaluate answer quality (e.g. `ragas`). |

## 📌 Quick Reference (cheat-sheet)

```text
Recall@K     = |relevant ∩ top_k| / |relevant|     # primary metric — track Recall@5/@10
Precision@K  = |relevant ∩ top_k| / k              # noise in the top-k
MRR          = mean(1 / rank_of_first_relevant)    # is the best chunk near the top?
Hit Rate     = mean(1 if any relevant in top_k)    # coarse yes/no
Ground truth = synthetic QA: LLM writes a question each chunk answers -> (question, chunk_id)
Tooling:     ragas (context_recall, context_precision, faithfulness, answer_relevancy)
Workflow:    build eval set -> measure -> change ONE thing -> re-measure -> keep if numbers rise
```

## 🛑 STOP — Self-Check

**Question:** After a chunking change, **Recall@5 rises 0.62 → 0.91** but users say answers are no
better. Where do you look next?

<details>
<summary>Answer</summary>

Retrieval is now good (the relevant chunk is in the top-5 91% of the time), so the remaining problem is
**generation**, not retrieval. Check: the chunks may be retrieved but **too noisy/low-precision** (add
**reranking**, Module 2.3, or raise Precision@K); the **prompt** may not force grounding ("answer only
from context"); the chunks may be retrieved but **truncated/lost-in-the-middle**; or the **answer
quality** itself needs measuring (faithfulness/answer-relevancy via `ragas`). The point: you've proven
it's *not* a retrieval problem, so stop tuning retrieval.
</details>

## 🎯 Interview angle

*"How do you know your RAG is good?"* — answer **eval-driven**: a synthetic QA set, **Recall@K** for
retrievability, **MRR** for rank, then answer-quality metrics (`ragas`) for generation, all run as
**regression tests** in CI. Stating that you **measure retrieval and generation separately** is the
senior signal interviewers listen for.
