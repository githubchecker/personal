### Module 0.0.1 — What Is an AI Model? (Parameters & Weights)

*Every conversation about AI costs, tradeoffs, and architecture decisions traces back to this.*

**Topics:**

- **What a parameter actually is** `[MUST KNOW]`
    - An LLM is a mathematical function stored as a file of numbers — nothing more
    - Each **parameter** (also called a **weight**) is a single floating-point number (e.g., `0.347`)
    - These numbers encode everything the model knows: grammar, facts, reasoning patterns, language structure
    - When you download Llama 3.1 8B, you are downloading a ~16GB file of 8 billion such numbers
    - **The knob analogy (copy to personal notes):** Imagine a mixing board with 7 billion knobs. Each knob is a parameter. Before training, all knobs are random. Training is the process of twisting every knob until the output sounds correct. Fine-tuning is gently adjusting a small subset of knobs. RAG is not touching any knob at all — just changing what you say into the microphone.
    - **Practical impact:** This is why a 70B model is smarter but slower and more expensive — 10× more knobs means 10× more computation per word generated
- **Why model size matters: 7B vs 70B vs 400B+** `[MUST KNOW]`
    - **7B models (Llama 3.2 8B, Phi-4, Mistral 7B):** Fast, cheap, runs on a laptop GPU. Good for classification, simple Q&A, routing decisions in agents
    - **70B models (Llama 3.1 70B, Qwen2.5 72B):** Strong reasoning, follows complex instructions. Needs server GPU or cloud API. The enterprise workhorse tier
    - **400B+ models (GPT-4o, GPT-4.1, Claude 3.7 Sonnet, Llama 3.1 405B):** Frontier reasoning, multimodal, best for complex agentic tasks. API-only for most teams
    - **Architect decision:** Cost-routing (7B for simple tasks, 70B for complex ones, 400B for frontier reasoning) is a core LLMOps skill covered in Phase 4
- **Model formats and quantisation basics** `[MUST KNOW]`
    - Common formats: `.safetensors` (Hugging Face standard), `GGUF` (for local CPU/GPU inference with Ollama)
    - **Quantisation:** Full precision (float32) = 4 bytes/parameter. Half (float16) = 2 bytes. Q4_K_M (GGUF) ≈ 0.5 bytes
    - A 7B model in Q4 ≈ 4GB — runnable on a MacBook Pro with 16GB RAM
    - **Practical impact:** When Phase 1 uses Ollama to run local models, this explains why `llama3.2:3b` loads in seconds but `llama3.1:70b` requires a machine with 48GB RAM
- **Model vs API — closed vs open-weight** `[MUST KNOW]`
    - **Closed APIs:** OpenAI (GPT-4o, GPT-4.1), Anthropic (Claude 3.7 Sonnet), Google (Gemini 2.5 Pro), Azure OpenAI — you never see the weights. Data leaves your infrastructure on every call
    - **Open-weight models:** Llama 3.x (Meta), Mistral, Phi-4 (Microsoft), Qwen2.5 (Alibaba) — downloadable, self-hostable. Data stays in your infrastructure
    - **Enterprise tradeoff:** Closed = better quality, easier, privacy risk. Open-weight = slightly lower quality, operational overhead, full data control
    - **When enterprises choose open-weight:** Healthcare (HIPAA), finance (PCI data), legal (client confidentiality), government (data sovereignty)
    - **Your Azure angle:** Azure OpenAI gives you GPT-4o with data residency guarantees. Azure AI Foundry lets you deploy Llama/Mistral/Phi as managed endpoints — covered in Phases 1 and 5

**No milestone project. Read, understand, take notes on the knob analogy.**

### Module 0.0.2 — How AI Got Here (Model Evolution Timeline)

*You will be asked “why transformers?” in architect interviews. This is the answer.*

**Topics:**

- **Pre-2017: RNNs and LSTMs — the old sequential approach** `[AWARENESS ONLY]`
    - RNNs processed text word by word, left to right — sequential and slow
    - The problem: they forgot early words by the end of a long sentence (“vanishing gradient problem”)
    - LSTMs partially solved this with a memory cell but remained sequential — impossible to parallelise
    - **Verdict: safely ignore for your work. No modern production LLM uses RNNs.** This section exists so you can say “I know why we moved away from RNNs” in an interview
- **2017: “Attention Is All You Need” — the Transformer revolution** `[MUST KNOW — conceptual only]`
    - Published by Google Brain in 2017 — the paper that changed everything
    - Core insight: instead of processing words sequentially, look at all words simultaneously and compute how much each word should “attend to” every other word
    - Example: “The bank by the river overflowed” — the model needs to attend “bank” to “river” (not “money”). Attention computes this automatically
    - **Why transformers won:** (1) Parallelisable across thousands of GPUs. (2) Long-range dependencies without degradation. (3) Scales with data and compute — bigger = smarter
    - **Practical impact:** Every model you use in Phases 1–4 is a transformer. Knowing this explains why context windows exist and why hallucination happens
- **2018–2022: BERT and GPT series — the family split** `[AWARENESS ONLY]`
    - **BERT (2018, Google):** Encoder-only transformer. Understands text. Used for embeddings, search, classification
    - **GPT-1/2/3 (2018–2020, OpenAI):** Decoder-only transformer. Generates text. The ChatGPT ancestor
    - **GPT-3 (2020):** 175B parameters — first model to show emergent reasoning at scale
    - **InstructGPT (2022):** GPT-3 + RLHF (human feedback). This became ChatGPT
    - **Key takeaway:** The field split into two model families — encoders for understanding, decoders for generating
- **2022–2026: ChatGPT to Agentic AI — the current era** `[MUST KNOW]`
    - **Nov 2022:** ChatGPT released — 100M users in 2 months. The enterprise AI budget era begins
    - **2023:** GPT-4, Claude 1/2, Llama 1/2 (Meta open-sources weights), Mistral 7B. Open-weight ecosystem begins
    - **2024:** GPT-4o (multimodal), Claude 3 family, Llama 3, Gemini 1.5 Pro (1M context), RAG becomes enterprise standard, LangGraph released
    - **2025:** Reasoning models (OpenAI o1/o3, DeepSeek-R1), MCP protocol released by Anthropic, LangGraph 0.2 enterprise features, agentic frameworks mature
    - **2026 (now):** Agentic AI is mainstream. Framework landscape explodes: OpenAI Agents SDK, Google ADK, Microsoft Agent Framework (AutoGen + Semantic Kernel unified, GA April 2026), CrewAI hits 44K+ GitHub stars. MCP crosses 200 server implementations. The skills in Phases 2–4 are what enterprises are actively hiring for right now

**No milestone project. This is a 45-minute read.**

### Module 0.0.3 — Encoder vs Decoder (The Architecture Split That Matters for RAG)

*This is the most practically important conceptual module. It determines which model you use at each stage of a RAG pipeline.*

**Topics:**

- **Encoder models — the “understanding” family** `[MUST KNOW]`
    - **What they do:** Read text and convert it into a dense numerical vector (embedding) that captures meaning
    - **Architecture:** Bidirectional — they see all words in both directions simultaneously
    - **Key models:** BERT, RoBERTa, `text-embedding-3-large` (OpenAI), `bge-large-en-v1.5` (BAAI), `nomic-embed-text`
    - **What they cannot do:** Generate new text. They are readers, not writers
    - **Where you use them:** Phase 2 RAG — every document chunk embedded into pgvector or Pinecone uses an encoder
    - **Code:** `from sentence_transformers import SentenceTransformer; model = SentenceTransformer('BAAI/bge-large-en-v1.5')` — this is an encoder
- **Decoder models — the “generation” family** `[MUST KNOW]`
    - **What they do:** Take a sequence of tokens and predict the next most likely token, one at a time
    - **Architecture:** Unidirectional (left to right) — each token can only attend to previous tokens
    - **Key models:** GPT-4o, GPT-4.1, Claude 3.7 Sonnet, Llama 3.1, Mistral, Phi-4, Qwen2.5, Gemini 2.5 Pro
    - **Where you use them:** Every LLM API call in Phases 1–4. The model that reads retrieved RAG chunks and generates an answer is always a decoder
    - **The generation loop:** Token 1 → Token 2 → Token 3 → … → `<END>`. Each token is a separate model forward pass. This is why streaming exists
- **Encoder-Decoder models — the “translation” family** `[AWARENESS ONLY]`
    - **Key models:** T5, BART, mT5, MarianMT
    - **Use cases:** Machine translation, document summarisation, question generation
    - **Verdict:** You will not build with these directly. Know they exist; do not study them
- **The practical mapping for your RAG pipeline** `[MUST KNOW]`
    - **Ingestion time:** Use an **encoder** model → convert document chunks → embedding vectors → store in vector DB
    - **Query time:** Use the **same encoder** → convert user question → query vector → retrieve similar chunks. Then use a **decoder** (LLM) → read retrieved chunks → generate answer
    - **The two-model architecture:** Every RAG system has at minimum one encoder and one decoder working together
    - **Azure-specific mapping:**
        - Encoder: `text-embedding-3-large` via Azure OpenAI (deployment type: Embedding)
        - Decoder: `gpt-4o` via Azure OpenAI (deployment type: Chat Completion)
    - **Critical rule:** Always use the same embedding model for both ingestion and query. Switching models means the vector space does not align and retrieval quality degrades

**No milestone project. Draw a diagram: PDF → Encoder → Vector DB → Encoder (query) → Decoder → Answer.**

### Module 0.0.4 — The Model Lifecycle (Who Does What — The Career-Defining Distinction)

*This answers “what is your job as an AI Engineer” and “why RAG instead of fine-tuning?” — both are standard architect interview questions.*

**Topics:**

- **Stage 1: Pre-training — what the big labs do** `[MUST KNOW — conceptual only]`
    - **Who:** Meta, OpenAI, Google, Anthropic, Mistral, Microsoft
    - **What:** Train on trillions of tokens of internet text by predicting the next token repeatedly. Every wrong prediction triggers a weight update across all parameters
    - **Cost:** $50M–$100M+. Thousands of H100 GPUs running for months
    - **Result:** A “base model” — knows language and facts but cannot follow instructions
    - **Your role:** Zero. You use the output of this stage
- **Stage 2: SFT (Supervised Fine-Tuning) — teaching manners** `[MUST KNOW — conceptual only]`
    - **Who:** The same labs, before public release
    - **What:** 50,000–500,000 human-written (question, ideal answer) pairs train the model to follow instruction format
    - **Result:** An “instruct” model — Llama 3.1 8B Instruct vs Llama 3.1 8B Base are the same weights post-SFT
    - **Your role:** Zero for major models. If you build a domain fine-tune in Phase 4, you run a smaller version of this
- **Stage 3: RLHF / DPO — safety and alignment** `[AWARENESS ONLY]`
    - **Who:** The labs, using human raters
    - **What:** Humans rank model outputs. The model learns to prefer highly-ranked (safe, helpful) outputs via reinforcement learning (RLHF) or direct preference optimisation (DPO)
    - **Result:** The final public “chat” model — ChatGPT, Claude, Gemini as you know them
    - **Constitutional AI:** Anthropic’s variant where an AI, not humans, provides feedback using a written “constitution” of principles
    - **Your role:** Zero for production models. Architect-level awareness only
- **Stage 4: Fine-Tuning / LoRA / PEFT — your customisation layer** `[MUST KNOW]`
    - **Who:** ML Engineers, AI Engineers — you, occasionally
    - **What:** Take a trained chat model and further train it on your specific data to adjust weights toward your domain
    - **LoRA (Low-Rank Adaptation):** Freezes original weights. Adds tiny trainable matrices. Trains only these. Cost: 1/1000th of full fine-tuning. Industry standard
    - **QLoRA:** LoRA + quantised base model. Runs on a single consumer GPU
    - **When to fine-tune vs RAG:** Fine-tune for style/tone/format. Use RAG for knowledge/facts. Fine-tuning does not reliably inject new factual knowledge — it changes behaviour, not memory
    - **Your role in Phase 4:** Conceptual awareness + knowing when to recommend it. Hands-on is optional
- **Stage 5: RAG and Agents — your primary work** `[MUST KNOW]`
    - **Who:** AI Application Engineers, AI Architects — you
    - **What:** Model weights are completely frozen. You write application code connecting a frozen LLM to external data and tools
    - **RAG:** Retrieve relevant documents → inject into context window → model answers from retrieved context. Zero parameter changes
    - **Agents:** LangGraph code reads model output → executes tools → feeds results back → loops until task complete. Still zero parameter changes
    - **The key insight (copy to notes):** As an AI Lead/Architect, you almost never touch model weights. Your job is the application layer around frozen models — the retrieval system, orchestration logic, evaluation pipeline, cost governance. That is 95% of enterprise AI engineering
    - **Cost comparison:** Pre-training ($50M+) → Fine-tuning ($500–$5,000) → RAG (API call cost, $0.001–$0.01/query)
- **The lifecycle cheat sheet — who touches the parameters (copy this to notes)** `[MUST KNOW]`
    
    
    | Stage | What happens to weights | Who does it | Cost |
    | --- | --- | --- | --- |
    | **Pre-training** | ALL parameters updated trillions of times | Meta / OpenAI / Google | $50M+ / months |
    | **SFT** | ALL parameters updated on curated Q&A pairs | The same labs | $1M+ / weeks |
    | **RLHF / DPO** | ALL parameters updated on human preference rankings | The same labs | $500K+ / weeks |
    | **LoRA / PEFT** | SMALL adapter matrices trained; originals frozen | ML Engineers / You (Phase 4) | $500–$5,000 / hours |
    | **RAG** | ZERO parameters changed. Context window injection only | AI Engineers / You | API call cost / instant |
    | **Agents** | ZERO parameters changed. Python orchestration only | AI Engineers / You | API call cost / instant |

**No milestone project. Internalise the cheat sheet table.**

### Module 0.0.5 — Core Vocabulary (The Language of Every AI Conversation)

*These terms appear in every JD, every architecture discussion, every vendor conversation.*

**Topics:**

- **Tokens — the atoms of language** `[MUST KNOW]`
    - LLMs do not read words. They read **tokens** — fragments produced by a tokeniser
    - Common words = 1 token. Long/uncommon words = 2–4 tokens. Numbers and code tokenise differently
    - **Rough conversion:** 1,000 tokens ≈ 750 words ≈ 1.5 pages of A4
    - **Why it matters for cost:** Every API charges per 1 million tokens (input + output). A single GPT-4o call processing a 10-page document ≈ 8,000 tokens = ~$0.02. At 100,000 daily calls → $2,000/day
    - **Why it matters for architecture:** Token counting drives every chunking decision in Phase 2, every context window management decision in Phase 3, and every cost optimisation decision in Phase 4
- **Context Window — the model’s working memory** `[MUST KNOW]`
    - The maximum number of tokens the model can process in a single call (input + output combined)
    - **Current standards:** GPT-4o = 128K, Claude 3.7 Sonnet = 200K, Gemini 2.5 Pro = 1M
    - **The “lost in the middle” problem:** Models perform worse on information buried in the middle of a long context. This is why chunking + retrieval (RAG) often beats stuffing everything into the context window
    - **Architectural constraint:** In Phase 3 LangGraph agents, every message in conversation history consumes tokens. Long-running agents need sliding-window or summarisation strategies to prevent context overflow
    - **Formula:** `system prompt + conversation history + retrieved chunks + output ≤ max context window`
- **Temperature and Top-P / Top-K — the output control dials** `[MUST KNOW]`
    - **Temperature:** Controls how “random” the model’s token selection is
        - Temperature 0: Always picks the highest-probability token. Deterministic, reproducible
        - Temperature 0.3–0.7: Balanced — some variation, still mostly accurate
        - Temperature 0.7–1.0: Creative — varied, sometimes surprising
        - **Agent rule:** Temperature = 0 on routing/supervisor nodes. Temperature 0.7 on final creative output nodes only
    - **Top-P (Nucleus Sampling):** Instead of picking from all possible tokens, restrict to the smallest set of tokens whose combined probability exceeds P (e.g., 0.9 = top 90% of probability mass)
        - Low Top-P (0.1): Very conservative, repetitive
        - High Top-P (0.95): More diverse vocabulary
        - **Practical guidance:** Temperature and Top-P interact — most providers recommend adjusting one at a time, not both simultaneously
    - **Top-K:** Restricts selection to the K highest-probability tokens at each step (e.g., Top-K=50 means choose from the 50 most likely next tokens)
        - Lower K = more focused/repetitive. Higher K = more creative/varied
        - Less commonly tuned than Temperature or Top-P in enterprise settings
    - **Interview answer:** “For agent routing nodes I use Temperature=0, Top-P=1 for fully deterministic routing. For final response generation I use Temperature=0.4, Top-P=0.9 for natural but consistent output.”
- **System / User / Assistant Roles — the chat structure** `[MUST KNOW]`
    - Modern LLMs expect structured conversation, not raw strings
    - **System role:** “God Mode” instruction — sets persona, constraints, behaviour. Treated with highest authority
    - **User role:** The human’s message
    - **Assistant role:** The model’s previous responses — this is how agents have memory, by feeding prior turns back
    - **In LangChain/LangGraph:** These map directly to `SystemMessage`, `HumanMessage`, `AIMessage` objects. Every LLM call in Phases 1–3 uses this structure
    - **Security implication:** Prompt injection attacks attempt to override the system role. System prompt hardening is a Phase 4 topic
- **Hallucination — why models confidently lie** `[MUST KNOW]`
    - LLMs predict the statistically most likely next token — they do not retrieve facts from a database
    - When training data on a specific fact is sparse, the model fills the gap with a plausible-sounding fabricated response — stated with full confidence
    - **Why it happens:** The training objective was “predict the next token,” not “be factually accurate”
    - **Types:** Factual (wrong date, wrong number), source (cites a paper that does not exist), reasoning (logically coherent but wrong conclusion)
    - **How RAG mitigates it:** Injecting actual source text into the context window gives the model ground truth to work from — this is “grounding”
    - **How evaluation catches it:** Ragas Faithfulness metric (Phase 4) measures whether the model’s answer is supported by retrieved context — this is automated hallucination detection
- **Grounding and the three types of AI memory** `[MUST KNOW]`
    - **Parametric memory:** Knowledge baked into weights during training. Static, cuts off at training date, cannot be updated without retraining
    - **In-context memory:** Information injected into the context window at query time. Temporary — exists only for that call. RAG uses this
    - **External memory:** Databases, vector stores, Redis. Persistent across sessions. Agents write to and read from these
    - **Grounding:** Connecting model outputs to verifiable, retrievable sources. A grounded response traces back to a specific document chunk
    - **The architect’s mental model:** RAG = replacing parametric memory with in-context memory for factual queries. Agents = combining all three memory types

**No milestone project. These terms should be in your working vocabulary before writing your first line of Python.**

### Module 0.0.6 — The AI Landscape (Types of Models and When to Use Each)

*Know the map to avoid recommending the wrong tool class in an architecture review.*

**Topics:**

- **Discriminative AI vs Generative AI** `[MUST KNOW]`
    - **Discriminative AI (classic ML):** Classifies or labels existing data. Input → Label. Spam filter, fraud detection, image classification. Does not create anything new
    - **Generative AI:** Creates new content from a prompt — text, images, code, audio
    - **Where classic ML still wins:** Anomaly detection in logs, fraud scoring, recommendation ranking, time-series forecasting. An XGBoost model beats an LLM on cost, latency, and accuracy for these
- **Foundation Models vs Task-Specific Models** `[MUST KNOW]`
    - **Foundation model:** Large model trained on broad data, adaptable to many tasks. GPT-4o, Claude, Llama 3.1
    - **Task-specific model:** Fine-tuned for one narrow job. Faster, cheaper, more accurate for that one task — useless for anything else
    - **Architect decision:** Foundation model for broad, varied tasks. Task-specific for high-volume, latency-sensitive, narrow tasks (classifying 10,000 support tickets/day)
- **Closed models vs Open-weight models** `[MUST KNOW]`
    - **Closed:** GPT-4o, GPT-4.1 (OpenAI), Claude 3.7 Sonnet (Anthropic), Gemini 2.5 Pro (Google) — API only, weights never shared, best performance, data leaves your infrastructure
    - **Open-weight:** Llama 3.x (Meta), Mistral, Phi-4 (Microsoft), Qwen2.5 (Alibaba) — downloadable, self-hostable, full data control
    - **Your Azure angle:** Azure OpenAI = closed model with data residency. Azure AI Foundry = open-weight models as managed endpoints. Both covered in Phases 1 and 5
- **Multimodal models** `[ARCHITECT AWARENESS]`
    - **Key models:** GPT-4o, GPT-4.1 (text + image), Claude 3.7 Sonnet (text + image), Gemini 2.5 Pro (text + image + audio + video), Phi-4-Vision, LLaVA
    - **Practical use in your roadmap:** Phase 2 multimodal RAG uses GPT-4o vision to read document page images — bypassing OCR for complex financial charts
    - **What you do NOT need:** Audio/video model training, diffusion model internals. Separate specialisations
- **Reasoning models vs standard chat models** `[ARCHITECT AWARENESS]`
    - **Standard chat (GPT-4o, Claude 3.7):** One forward pass. Fast (1–5 seconds). Good for most tasks
    - **Reasoning models (OpenAI o3, DeepSeek-R1, Claude with extended thinking):** Internal chain-of-thought before responding. Slow (10–120 seconds). Much better at complex multi-step problems
    - **Cost:** Reasoning models cost 5–20× more. Use only where standard models demonstrably fail
    - **Architect decision:** Route complex reasoning tasks (multi-hop analysis, code architecture review) to reasoning models. Simple queries go to standard models. This routing logic is Phase 4

**No milestone project. After this module you have enough vocabulary to read any AI engineering blog post, JD, or architecture document without stopping to look things up.**
